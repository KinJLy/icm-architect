# NDIS High Cost AT Report — structure

The report is organised into seven blocks. Follow this order every time; don't reinvent it per case.

1. **Introduction** — cover page, table of contents, executive summary.
2. **Participant Story** — disability impact, living environment, alignment with NDIS goals.
3. **Clinical Justification** — assessment findings, alternative options considered, reasoning for the recommended AT.
4. **Clinical Trial Evidence** — in-clinic trial + community trial results, including washout findings. (Skip if the case has no device trial.)
5. **Supporting Evidence** — research summary, risk assessment, cost-benefit analysis.
6. **Funding Justification** — Section 34 Reasonable & Necessary criteria, final recommendation.
7. **Appendices** — raw evidence only (see below).

A full submission runs roughly 12–20 pages of body text (excluding appendices).

## The central argument, stated once

Frame the recommendation around **demonstrated benefit, not projected benefit**, wherever trial data exists:

> "This recommendation is not based on projected benefit. It is based on demonstrated benefit observed through clinical assessment, community trial, objective outcome measures and washout findings."

Where a device was trialled, the strongest structure is: **baseline (problem) → trial (clear improvement) → washout (deterioration on withdrawal)**. This pattern is hard for an assessor to argue against because it isolates the device as the cause, not an incidental correlation. Don't bury the washout comparison — it's usually the single strongest piece of evidence in the report.

Low prior fall incidence is not evidence of safety — reframe it as evidence of activity restriction and behavioural compensation. Don't let "they haven't fallen much" read as "they don't need this."

## Appendices — evidence only, no summaries

**Rule:** the report body (Blocks 1–6) already contains the clinical interpretation, findings, and reasoning in full. Appendices exist to hold the *original supporting documents* an assessor might want to check, not a second write-up of the same content. If you catch yourself drafting an "Assessment Summary" or "Trial Summary" document for the appendix, stop — that content belongs in the body, and the appendix should carry the raw evidence it's summarising instead.

Standard appendix set (confirmed against an actual near-final KOrthotics submission draft, not just chat guidance — adapt per case, drop any that don't apply):

| Appendix | Contents |
|---|---|
| A – Outcome Measure Evidence | A1 raw physiotherapy outcome-measure summary table (10MWT, TUG, 6MWT, DGI, 30STS); A2 FSS; A3 PSFS |
| B – Community Trial Documentation | B1 8-week community trial logbook; B2 3-week washout logbook (participant feedback recorded during washout can live inside this, not as a separate appendix); B3 device-generated usage analytics, if the device captures them (don't skip this — see `submission-compliance-checklist.md`) |
| C – Participant Feedback | Participant feedback statement (1–2 pages, direct quotes — not the full logbook a second time) |
| D – Carer Feedback | Carer/family feedback statement |
| E – Medical Supporting Documentation | Specialist (e.g. neurologist) letter |
| F – Financial Documentation | Supplier quotation |

No separate goals appendix — the NDIS goal statement is quoted directly in the body (Block 2/3), not repeated as an appendix.

Keep each appendix to the original document — don't re-summarise it in a cover paragraph beyond a one-line "Purpose" note.

## Submitting

Two ways to package, in order of preference:

1. **Two files** (preferred): `AT_Report_<device>.pdf` (report body only) + `Supporting_Evidence_<device>.pdf` (every appendix combined, in order). Easiest for an assessor — they only open two documents.
2. **Numbered files**: `01_AT_Report_<device>.pdf`, `02_Appendix_A_...pdf`, `03_Appendix_B_...pdf`, etc. — use only if the recipient has asked for appendices split out.

Email body should list every attachment by name so the assessor can confirm nothing's missing before opening anything.
